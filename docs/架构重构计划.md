# SISM 项目架构重构计划

## 概述

本文档针对 SISM (Strategic Indicator Management System) 项目中发现的架构问题，提供详细的重构方案和实施计划。

## 问题分析与解决方案

### 问题 1: indicator 与 strategic-indicator 职责重叠

**现状：**
- `src/features/indicator/` - 存在指标相关功能
- `src/features/strategic-indicator/` - 也存在指标相关功能
- 两个模块职责边界模糊，容易导致开发者困惑

**分析：**
- `indicator` 模块包含：api/, model/, ui/
- `strategic-indicator` 模块包含：api/, lib/, model/, ui/, README.md
- 从 README.md 看，strategic-indicator 是战略指标相关的完整功能模块
- indicator 模块相对较简单

**解决方案：合并两个模块**

**实施步骤：**
1. 保留 `strategic-indicator` 作为主模块（因为它更完整）
2. 将 `indicator` 中的代码合并到 `strategic-indicator` 中
3. 如果有功能差异，需要明确区分：
   - `indicator` 可能是通用指标功能
   - `strategic-indicator` 是战略指标专用功能
4. 如果确实是不同功能，考虑重命名：
   - `indicator` → `common-indicator` (通用指标)
   - `strategic-indicator` → `strategic-indicator` (战略指标)

**风险评估：**
- 低风险：只需移动文件和更新导入路径
- 需要确保所有引用都正确更新

### 问题 2: useIndicatorStatus.ts 位置不当

**现状：**
- 文件位置：`src/features/dashboard/model/useIndicatorStatus.ts`
- 根据 FSD (Feature-Sliced Design) 最佳实践
- Composable 应该放在 `composables/` 目录，而不是 `model/` 目录
- `model/` 层应该只包含纯粹的状态管理（如 Pinia store）和类型定义

**分析：**
- useIndicatorStatus.ts 是一个纯函数式的 composable
- 它包含状态计算逻辑，不是状态管理
- 文件顶部注释已经写了 `@module composables/dashboard`，说明开发者知道应该放在哪里

**解决方案：移动到正确位置**

**实施步骤：**
1. 创建目录：`src/composables/dashboard/`
2. 移动文件：`src/features/dashboard/model/useIndicatorStatus.ts` → `src/composables/dashboard/useIndicatorStatus.ts`
3. 更新所有导入路径
4. 验证功能正常

**需要更新的导入：**
```typescript
// 旧路径
import { getIndicatorStatus } from '@/features/dashboard/model/useIndicatorStatus'

// 新路径
import { getIndicatorStatus } from '@/composables/dashboard/useIndicatorStatus'
```

**风险评估：**
- 低风险：只需移动文件和更新导入
- 使用 IDE 的重构工具可以自动处理

### 问题 3: 顶级目录过于宽泛

**现状：**
在 `src/` 下直接创建了以下目录：
- `api/`
- `composables/`
- `config/`
- `constants/`

**问题分析：**
根据 FSD 架构最佳实践，这些目录应该归类到 `shared/` 层，而不是直接放在 `src/` 下。FSD 鼓励将共享代码按照其所属的业务或共享层级进行归类。

**当前结构问题：**
```
src/
├── api/              ❌ 应该在 shared/api/
├── composables/      ❌ 应该在 shared/composables/
├── config/           ❌ 应该在 shared/config/
├── constants/        ❌ 应该在 shared/constants/
├── app/
├── pages/
├── widgets/
├── features/
├── entities/
└── shared/           ✅ 正确的共享层位置
```

**解决方案：归入 shared 层**

**目标结构：**
```
src/
├── app/
├── pages/
├── widgets/
├── features/
├── entities/
└── shared/
    ├── api/          ✅ 移动后
    ├── composables/  ✅ 移动后
    ├── config/       ✅ 移动后
    └── constants/    ✅ 移动后
```

**实施步骤：**

1. **移动 api/ 目录**
   - 移动：`src/api/` → `src/shared/api/`
   - 更新所有导入路径
   - 更新相关配置

2. **移动 composables/ 目录**
   - 移动：`src/composables/` → `src/shared/composables/`
   - 更新所有导入路径
   - 与问题 2 中的 useIndicatorStatus.ts 一起处理

3. **移动 config/ 目录**
   - 移动：`src/config/` → `src/shared/config/`
   - 更新所有导入路径

4. **移动 constants/ 目录**
   - 移动：`src/constants/` → `src/shared/constants/`
   - 更新所有导入路径

**导入路径更新示例：**
```typescript
// 旧路径
import { apiClient } from '@/api/client'
import { useSomeComposable } from '@/composables/useSomeComposable'
import { appConfig } from '@/config/app'
import { CONSTANTS } from '@/constants'

// 新路径
import { apiClient } from '@/shared/api/client'
import { useSomeComposable } from '@/shared/composables/useSomeComposable'
import { appConfig } from '@/shared/config/app'
import { CONSTANTS } from '@/shared/constants'
```

**风险评估：**
- 中风险：涉及大量文件移动和导入更新
- 需要仔细测试确保所有功能正常
- 建议使用 Git 分支进行重构，方便回滚

### 问题 4: entities/milestone/api 位置不当

**现状：**
- 文件位置：`src/entities/milestone/api/milestoneApi.ts`
- Entities 层在 FSD 中应该只包含最核心、最纯粹的业务实体模型和类型定义
- API 调用属于 Features 层或 Shared/API 层

**分析：**
- Entities 层应该是纯数据模型，不包含副作用
- API 调用是业务逻辑的一部分，应该放在 Features 层
- 或者如果是通用的 API，可以放在 Shared/API 层

**解决方案：移动到正确位置**

**方案 A：移动到 Features 层（推荐）**
- 移动：`src/entities/milestone/api/` → `src/features/milestone/api/`
- `src/entities/milestone/` 只保留 `model/`

**方案 B：移动到 Shared 层**
- 如果 API 是通用的，移动到 `src/shared/api/milestone/`

**实施步骤：**

1. 检查 API 的使用范围
2. 如果只在 milestone 相关功能中使用，采用方案 A
3. 如果是通用 API，采用方案 B
4. 移动文件并更新导入路径

**Entities 层应该只包含：**
```
src/entities/milestone/
├── index.ts
└── model/
    ├── types.ts      # 类型定义
    └── schema.ts     # Zod schema（如果有）
```

**风险评估：**
- 低风险：只需移动文件和更新导入
- 需要检查 API 的使用范围

## 重构实施计划

### 阶段 1: 低风险重构（问题 1 和 2）

**目标：** 解决职责重叠和 composable 位置问题
**时间估计：** 2-3 小时
**风险：** 低

**步骤：**
1. 合并 indicator 和 strategic-indicator 模块
2. 移动 useIndicatorStatus.ts 到正确位置
3. 测试所有功能正常
4. 提交代码

### 阶段 2: 中风险重构（问题 4）

**目标：** 解决实体层 API 位置问题
**时间估计：** 1-2 小时
**风险：** 低

**步骤：**
1. 移动 entities/milestone/api 到正确位置
2. 更新导入路径
3. 测试相关功能
4. 提交代码

### 阶段 3: 高风险重构（问题 3）

**目标：** 重构顶级目录结构
**时间估计：** 4-6 小时
**风险：** 中

**步骤：**
1. 创建 Git 分支 `refactor/fsd-directory-structure`
2. 逐个移动目录
3. 使用 IDE 的重构工具批量更新导入
4. 全面测试
5. 提交代码（保留回滚能力）
