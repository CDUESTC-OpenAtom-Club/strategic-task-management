# 审批中心产品稿初稿

## 文档信息

- 文档名称：审批中心统一产品稿初稿
- 适用范围：前端全部审批中心组件
- 当前阶段：初稿
- 输出目标：统一审批中心中的详情查看、审批判断、历史追溯与材料查看体验
- 当前落地顺序：先在 `DistributionApprovalProgressDrawer` 上验证“本次上报内容”模块，再推广到其他审批中心组件

## 一、背景

当前系统中的审批中心已经具备基础流程能力，用户可以看到：

- 待审批实例
- 审批流程
- 审批历史
- 查看详情
- 审批通过 / 驳回

但审批人在实际使用时，仍存在明显断层：

1. 进入审批中心后，可以看到流程，却不一定能直接看到本次到底提交了什么业务内容。
2. “查看详情”更多停留在流程实例层，缺少业务填报层的内容快照。
3. 对于带附件的上报、下发、填报记录，审批人往往无法在同一处完成材料核查。
4. 历史审批实例虽然能查看流程记录，但难以回看当时到底审批了哪些业务内容。

这会直接影响：

- 审批效率
- 审批准确性
- 历史追溯能力
- 后续复盘能力

## 二、产品目标

审批中心统一升级后的目标是：

1. 审批人进入任一审批中心组件后，不离开当前上下文即可完成审批判断。
2. “查看详情”必须同时承载流程信息和业务内容，而不是只有流程信息。
3. 每一笔审批实例，在进入审批历史后，仍可回看当次审批所对应的业务快照。
4. 带附件的审批内容必须支持在审批详情中直接查看材料。

## 三、适用组件范围

本稿适用于全部审批中心相关组件，而不是只针对某一个页面。

当前应统一纳入规范的组件包括：

- `ApprovalProgressDrawer.vue`
- `DistributionApprovalProgressDrawer.vue`
- `PlanApprovalDrawer.vue`
- 其他后续复用审批中心详情结构的审批抽屉 / 审批弹窗

统一要求：

- 信息区块结构统一
- 标题文案统一
- “查看详情”能力统一
- 附件查看能力统一
- 历史回溯逻辑统一

## 四、统一信息架构

所有审批中心详情视图统一采用三段式结构：

1. 实例信息
2. 本次业务内容
3. 审批历史

其中：

- 实例信息回答“这是一笔什么审批”
- 本次业务内容回答“这次到底提交了什么”
- 审批历史回答“这笔审批是怎么流转的”

不允许再出现只有流程、没有业务内容的“空壳详情页”。

## 五、实例信息区设计

实例信息区用于展示审批实例本身的识别信息。

建议字段：

- 实例编号
- 审批标题
- 审批类型
- 提交人
- 提交部门
- 提交时间
- 当前步骤
- 当前状态
- 关联实体 ID
- 流转方向

示例：

- 上报审批：继续教育部 -> 战略发展部
- 下发审批：战略发展部 -> 继续教育部

设计要求：

- 实例信息区保持简洁，不承载业务明细
- 审批状态必须明显可见
- 当前步骤必须明确显示

## 六、本次业务内容区设计

这是审批详情中的核心区域。

### 6.1 区域定位

该区域放置在“实例信息”和“审批历史”之间，用于向审批人展示：

- 本次上报了什么
- 本次下发了什么
- 本次填报了什么
- 本次修改了什么
- 本次提交了哪些材料

### 6.2 区域标题建议

按业务类型统一命名，不使用过于抽象的标题。

推荐标题：

- `本次上报内容`
- `本次下发内容`
- `本次填报内容`
- `业务内容快照`

推荐副标题：

- `以下内容为发起审批时提交的业务明细`

### 6.3 展示原则

本次业务内容区必须满足三个原则：

1. 以审批判断为中心，不以字段堆叠为中心
2. 默认先展示核心字段，再允许展开更多
3. 优先呈现“人要审批的内容”，而不是系统内部元数据

## 七、指标类审批内容设计

对于 `/indicators` 这类指标填报 / 指标上报场景，业务内容区应按“指标卡片列表”展示。

### 7.1 每个指标卡片的核心字段

必显字段：

- 指标名称
- 指标类型
- 责任部门
- 当前实际进度
- 本次填报进度
- 填报说明
- 提交材料

可选字段（目前没必要）：

- 目标值
- 实际值
- 单位
- 指标编号
- 年度
- 所属任务 / 所属计划

### 7.2 指标卡片推荐结构

每张卡片建议结构如下：

1. 卡片头部
   - 指标名称
   - 指标类型标签
   - 审批状态标签（如有需要）

2. 进度区
   - 当前实际进度
   - 本次填报进度

3. 说明区
   - 填报说明全文

4. 材料区
   - 附件列表
   - 点击查看 / 下载

5. 更多信息区
   - 目标值
   - 实际值
   - 单位
   - 责任部门

### 7.3 空态规则

- 无填报说明：显示 `未填写说明`
- 无附件：显示 `未提交附件`
- 无指标明细：显示 `本次审批未包含指标明细`

## 八、附件查看设计

附件查看是全部审批中心组件的通用能力，必须统一。

### 8.1 展示方式

附件在业务内容区中显示为附件列表。

展示要求：

- 显示真实文件名
- 不只显示“附件1、附件2”
- 附件较多时支持换行展示

### 8.2 点击行为

统一规则：

- 可预览文件：点击后新开页预览
- 不可预览文件：点击后下载

### 8.3 审批价值

附件区的目的不是“证明有附件”，而是让审批人能完成材料核查。

审批人在详情页里应能直接判断：

- 是否有材料
- 材料是否完整
- 材料是否符合说明内容

## 九、审批历史区设计

审批历史区用于展示流程动作，不替代业务内容区。

展示内容：

- 提交
- 通过
- 驳回
- 撤回
- 处理人
- 处理时间
- 审批意见

设计原则：

- 审批历史只解决“谁在什么时候做了什么”
- 不负责承载本次业务快照
- 审批意见和业务内容可以互相参照，但不能混写

## 十、查看详情的统一定义

以后审批中心中的“查看详情”必须统一理解为：

> 查看本次审批实例的完整业务上下文，而不是只看流程。

因此“查看详情”最少要包括：

1. 实例信息
2. 本次业务内容
3. 审批历史

如果只有实例信息和历史记录，没有业务内容，则视为详情能力不完整。

## 十一、待审批与审批历史的差异

同一套详情结构，在不同状态下有不同重点。

### 11.1 待审批实例

重点是“判断”。

审批人最关心：

- 这次到底提交了什么
- 附件是否齐全
- 填报说明是否合理
- 当前是否应该通过 / 驳回

### 11.2 历史审批实例

重点是“追溯”。

用户最关心：

- 当时审批的是什么
- 为什么通过 / 驳回
- 提交时附了哪些材料
- 当时的审批意见是什么

因此历史详情页必须尽量看到“当次快照”，而不是当前实时主数据。

## 十二、业务快照原则

业务快照分两阶段建设。

### 12.1 第一阶段：审批可用快照

目标：满足审批判断。

建议至少固化：

- 业务对象 ID
- 业务对象名称
- 本次填报进度
- 本次填报说明
- 本次提交附件

### 12.2 第二阶段：完整历史快照

目标：满足审计和复盘。

建议进一步固化：

- 类型
- 责任部门
- 目标值
- 实际值
- 单位
- 页面展示中会影响判断的其他关键字段

如果没有完整快照，历史详情就只能做到“近似回放”，做不到“完整还原”。

## 十三、组件统一要求

全部审批中心组件都需要满足以下统一规范：

1. 有详情就必须有业务内容区
2. 有附件就必须在详情中可查看
3. 有审批历史就必须可回看对应业务快照
4. 同类型审批在不同组件中的字段结构尽量统一
5. 不允许同一个审批中心中，一个入口能看业务内容，另一个入口只能看流程

## 十四、当前推荐落地顺序

### 第一步

统一详情结构：

- 实例信息
- 本次业务内容
- 审批历史

### 第二步

先把指标类审批内容落下来：

- 指标名称
- 填报进度
- 填报说明
- 附件

### 第三步

再逐步推广到其他审批业务：

- 计划审批
- 下发审批
- 任务审批
- 其他需要业务快照的审批中心视图

### 第四步

如需强历史追溯，再补完整快照字段

## 十五、验收标准

本稿对应的审批中心统一体验应满足以下验收标准：

1. 审批人点开任一审批实例详情后，可以直接看到“本次业务内容”。
2. 指标类审批中，审批人可以看到每个指标的填报进度、说明和附件。
3. 附件可直接点击查看或下载。
4. 审批历史中的历史实例也能查看当次业务内容快照。
5. 审批判断不需要再回到原始业务页面反复切换。
