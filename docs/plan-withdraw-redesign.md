# Plan 发起后可撤回设计方案

## 概述

本文档定义 `/strategic-tasks` 页面中“发起审批后，在审批前段允许发起人撤回”的目标设计，并同步约束共用审批展示逻辑在 `/indicators`、`/distribution` 页面中的一致性，重点覆盖：

- 页面按钮与页面状态如何变化
- 前端状态源与判断口径
- 后端撤回接口职责
- 数据库字段在撤回前后的变化
- 审批抽屉在“草稿 / 审批中 / 撤回后 / 再次发起”几个阶段的展示规则

本文档对应的典型场景是：

1. 战略发展部发起人误点“发起审批”
2. 当前审批流程仍处于允许撤回的前段节点
3. 发起人点击“撤回”
4. 页面恢复到与“尚未发起审批”一致的草稿编辑态

---

## 目标体验

以 `http://localhost:3500/strategic-tasks` 为准，目标行为如下：

### 1. 草稿态

- 顶部主按钮显示 `发起审批`
- 右侧审批入口显示 `审批进度`
- Plan 状态显示 `草稿`
- 指标可编辑、可删除、可新增
- 基础性权重校验仍按草稿态执行
- 打开审批抽屉时，沿用原有审批链展示，节点状态使用现有系统状态词

### 2. 已发起但仍可撤回

- 顶部主按钮显示 `撤回`
- Plan 状态显示 `待审批`
- 右侧审批入口显示 `审批中`
- 页面进入不可编辑态
- 只有在工作流允许撤回时，顶部 `撤回` 按钮可点击

### 3. 撤回成功后

- 顶部主按钮恢复为 `发起审批`
- 右侧审批入口恢复为 `审批进度`
- Plan 状态恢复为 `草稿`
- 页面恢复为可编辑态，表现与“最初未下发”一致
- 审批抽屉继续展示原有审批链
- 当前被撤回的节点前端显示为 `已撤回`

### 4. 撤回后再次发起

- 顶部主按钮仍为 `发起审批`
- 点击后不新建实例，直接复用原审批实例
- 原来最新的“已撤回”节点恢复为 `待审批`
- 页面重新进入 `待审批 / 审批中`

---

## 当前问题

当前实现存在 3 个口径混用问题：

### 问题 1. 撤回语义混用了 Plan 和 Indicator

页面顶部“撤回”本质上是“撤回当前 Plan 的审批申请”，不是逐条指标撤回，也不是先按指标状态筛一批数据再批量撤回。

但实现过程中曾混入逐条指标撤回逻辑，导致：

- 前端认为可以撤回
- 后端指标接口却报 `Cannot withdraw indicator: not in PENDING or DISTRIBUTED state`

根因是：

- `Plan` 的撤回接口是 `POST /api/v1/plans/{id}/withdraw`
- `Indicator` 的撤回接口是 `POST /api/v1/indicators/{id}/withdraw`
- 这两个接口的业务语义不同
- `/strategic-tasks` 页面顶部撤回的对象应始终是一整条 `Plan`
- `Indicator` 状态只是跟随 `Plan` 同步，不应反向决定页面是否允许撤回

### 问题 2. 页面状态和数据状态来源不统一

页面有多种状态来源：

- `plan.status`
- `plan.workflowStatus`
- `plan.canWithdraw`
- `indicator.status`

如果顶部按钮按 `Plan` 判，撤回候选又按指标局部状态判，就容易出现：

- 按钮可点
- 点击后又提示“没有可撤回指标”

而当前后端的真实实现口径是：

- 页面主状态来自 `PlanResponse`
- `workflowStatus / currentStepName / canWithdraw` 由 `PlanWorkflowSnapshotQueryService` 按工作流实例快照补齐
- `indicator.status` 由 `syncIndicatorStatusWithPlan()` 跟随 `Plan` 统一同步

因此页面不应再从指标层状态反推页面总态。

### 问题 3. 审批抽屉状态口径被改复杂了

审批抽屉当前最容易混淆的是“流程已经发起且仍在运行中”和“流程已经结束”这两种语义。

根据当前后端逻辑：

- 只要发起人点击了“发起审批”，审批实例就已经创建
- 此时流程就是运行中的，不是历史
- 即使当前只走到第一个节点，也仍然是运行中的审批实例
- 只有流程真正走到数据库中的最后一个结束节点，并通过，才算结束

如果把撤回后的抽屉错误改造成另一套展示逻辑，就会出现：

- 页面想回到草稿编辑态
- 抽屉却被改造成另一套不符合现有审批链的展示逻辑

这会造成明显的产品认知冲突。

---

## 设计原则

### 原则 1. 顶部撤回应始终是 Plan 级操作

`/strategic-tasks` 顶部“撤回”只对应：

- 当前部门当前年度的 `Plan`

不对应：

- 单条指标撤回
- 批量指标撤回
- 以 `statusAudit` 为基础的伪批量撤回

### 原则 2. 页面是否可编辑只看 Plan 口径

`/strategic-tasks` 页面所有编辑态、按钮态、标签态，统一按 Plan 口径判断：

- `status`
- `workflowStatus`
- `canWithdraw`

不要再用指标自身状态反推页面总态。

更具体地说：

- 页面以 `PlanResponse` 为唯一主状态源
- `plan.status` 表示业务状态
- `plan.workflowStatus` 表示当前审批实例状态
- `plan.canWithdraw` 表示当前审批实例是否仍处于允许发起人撤回的前段
- `indicator.status` 仅作为列表展示结果，不参与顶部按钮判定

### 原则 3. 审批抽屉沿用原有实例链路，不新增展示模式

- 审批抽屉不需要拆成多种展示模式
- 直接沿用当前系统已有审批链和数据库中的节点状态词
- 前端继续显示中文状态文案，数据库状态值只作为实现口径
- 只修正状态判断逻辑，不改 UI 设计语义

### 原则 4. “审批中”表示实例在运行，不表示已进入历史

- 发起人点击“发起审批”后，审批实例立即进入运行态
- 只要还没有走到最后一个结束节点并通过，就仍然属于“审批中”
- 第一个节点待处理时，不应被误判成“历史”
- 只要流程没有走到结束节点，就仍然是当前活跃实例

## 前端设计

除 `/strategic-tasks` 外，以下共用页面也必须跟随同一套状态口径：

- `http://localhost:3500/indicators`
- `http://localhost:3500/distribution`

要求：

- 共用审批抽屉中的节点状态映射保持一致
- 只要 Plan 进入 `PENDING` 审批中，页面编辑入口必须按 Plan 口径锁定
- 撤回成功后，恢复为可编辑态
- 再次发起后，再次进入不可编辑态

## 页面主状态机

建议把 `/strategic-tasks` 页面的顶部行为抽象为以下状态机：

| 页面态 | 判定口径 | 顶部主按钮 | 审批入口 | 是否可编辑 |
|------|---------|-----------|---------|-----------|
| Draft | `plan.status = DRAFT` 且无活动实例 | `发起审批` | `审批进度` | 是 |
| PendingWithdrawable | `plan.status = PENDING` 且 `workflowStatus = IN_REVIEW` 且 `canWithdraw = true` | `撤回` | `审批中` | 否 |
| PendingLocked | `plan.status = PENDING` 且 `workflowStatus = IN_REVIEW` 且 `canWithdraw = false` | `撤回(禁用)` | `审批中` | 否 |
| Distributed | `plan.status = DISTRIBUTED` | `撤回(禁用)` 或隐藏 | `查看审批` | 否 |

说明：

- 实际页面仍可保留现有按钮样式，但状态机要单一、可推导。
- `PendingWithdrawable` 是本设计最关键的新增确认态。
- 当前后端 `canWithdraw` 的计算口径是：
  - `workflowStatus = IN_REVIEW`
  - 且当前审批实例“已处理节点数 <= 1”
- 因此“可撤回”不是单纯看是否已发起，而是看是否仍停留在允许发起人撤回的前段节点。

## 页面数据源

前端页面只使用以下 Plan 字段作为主状态源：

| 字段 | 来源 | 用途 |
|------|------|------|
| `id` | `plans` 列表 / `currentPlan` | 标识当前 Plan |
| `status` | `plan.status` | 草稿 / 待审批 / 已下发 |
| `workflowStatus` | 工作流快照 | 是否在审批中 |
| `canWithdraw` | 工作流快照 | 是否允许撤回 |
| `currentStepName` | 工作流快照 | 当前审批节点展示 |
| `workflowInstanceId` | 工作流快照 | 是否存在活动审批实例 |

页面不再用以下信息判断顶部撤回资格：

- 任意指标级撤回字段
- `indicator.status` 的局部值
- 任何“从列表行状态反推页面态”的二次计算

这些字段最多用于列表展示，不作为 Plan 顶部撤回的准入条件。

## 顶部按钮行为

### 发起审批

触发条件：

- 当前页处于草稿态
- 权重校验通过
- 当前部门存在可提交指标

行为：

1. 调用提交 Plan 审批接口
2. 成功后刷新 `plan list`
3. 刷新当前年度指标列表
4. 页面切到 `PendingWithdrawable` 或 `PendingLocked`

### 撤回

触发条件：

- `plan.canWithdraw = true`

行为：

1. 调用 `POST /api/v1/plans/{id}/withdraw`
2. 刷新 `plan list`
3. 刷新当前年度指标列表
4. 页面恢复到 `Draft`

撤回成功后前端应立即同步以下 UI：

- 主按钮文本：`撤回 -> 发起审批`
- 审批入口文本：`审批中 -> 审批进度`
- 状态标签：`待审批 -> 草稿`
- 表格操作：恢复编辑
- 抽屉中当前节点前端显示改为 `已撤回`

注意：

- 顶部撤回不需要逐条调用指标撤回接口
- 指标状态应继续由后端随 `Plan` 一起同步
- 前端不应出现“先找可撤回指标，再决定能不能撤回 Plan”的逻辑

## 审批抽屉设计

审批抽屉继续沿用原来的设计，不新增模式，不重新发明节点状态词，只修正状态判断逻辑。

### 抽屉展示原则

- 抽屉始终展示同一条审批实例链路
- 只要流程没有到结束节点，该实例就仍然是活跃实例
- 即使当前还只在第一个节点，这条实例也不是历史
- 节点状态直接沿用数据库和原有系统中的状态词
- 不引入新的节点状态文案

### 撤回后的抽屉口径

- 撤回成功后，页面主状态恢复 `DRAFT`
- 抽屉不切换为另一套展示态
- 抽屉仍按原有实例链展示
- 当前最新节点前端显示为 `已撤回`
- 其他历史节点继续沿用原有系统状态口径
- 当前实例继续保留为活跃实例，但 `canWithdraw = false`

### 节点显示状态与审批权限的关系

- 节点展示状态和“当前是否允许审批”不是同一个概念
- 当前后端运行时只维护当前可推进节点，不会把后续审批节点同时激活成新的当前节点
- 撤回后，当前节点数据库状态改成 `WITHDRAWN`，前端显示为 `已撤回`
- 因此实例中不再存在可审批的 `PENDING` 节点
- 再次发起时，会复用原实例并把最新 `WITHDRAWN` 节点恢复成 `PENDING`，前端重新显示为 `待审批`
- 只有恢复后的当前节点重新通过，流程才会继续向下推进

### 运行态判定说明

- 发起审批后，只要实例已创建且未走到最终结束节点，就属于运行态
- 第一个节点刚创建、尚未审批时，也属于运行态
- 没有到结束节点前，这条实例一直是活跃的

## 与后端实现对齐后的结论

结合当前后端与数据库，方案口径应明确为：

1. `/strategic-tasks` 顶部撤回是整条 `Plan` 的撤回，不是指标撤回。
2. 页面总态以 `PlanResponse` 为准，不再从指标状态倒推。
3. 指标状态跟随 `Plan` 同步，页面是否可编辑、是否显示撤回、审批入口文案都看 `Plan`。
4. 审批抽屉中，“已发起但尚未到终态”的实例始终属于运行态，不属于历史。
5. 只有流程真正到达最后一个结束节点并通过，才能视为流程结束并进入历史。
6. 撤回成功后，当前页面回到草稿，抽屉仍沿用原有审批链展示，当前节点前端显示为 `已撤回`。
7. 审批不通过后，页面同样回到草稿，顶部主按钮仍显示 `发起审批`。
8. 再次发起时复用原审批实例，不重新创建一条新实例。

## 状态口径确认

本方案按你最新确认的页面口径收敛为三种主状态：

- `DRAFT`
- `PENDING`
- `DISTRIBUTED`

其中：

- 审批不通过后，页面恢复到 `DRAFT`
- 页面顶部主按钮恢复 `发起审批`
- 不再引入 `RETURNED` 独立页面态
- 不再引入 `重新提交` 这一套单独按钮语义

## 后端设计

## 撤回接口职责

顶部撤回只调用：

- `POST /api/v1/plans/{id}/withdraw`

职责：

1. 校验当前 Plan 是否允许撤回
2. 将 `plan.status` 改为 `DRAFT`
3. 同步当前 Plan 下指标状态回到草稿口径
4. 将当前审批实例中的最新 `PENDING` 节点改成 `WITHDRAWN`
5. 返回最新 Plan 快照

不应由前端自行循环调：

- `POST /api/v1/indicators/{id}/withdraw`

原因：

- 指标撤回接口是指标级语义
- 顶部按钮是 Plan 级语义
- 两者混用会导致业务错乱

## 工作流快照职责

后端 `PlanWorkflowSnapshotQueryService` 负责向前端返回：

- `workflowInstanceId`
- `workflowStatus`
- `currentStepName`
- `currentApproverId`
- `currentApproverName`
- `canWithdraw`

### 撤回后快照规则

撤回成功后，后端再次查询 Plan 快照时应满足：

- 实例状态仍落在当前审批实例口径内
- `canWithdraw = false`
- `workflowStatus` 仍按当前实例真实状态返回
- 当前实例中最新被撤回的节点仍可作为抽屉当前展示节点返回
- 因此 `currentStepName / currentApproverName` 可以继续保留最近节点信息
- 历史节点仍由抽屉中的审批链查询负责展示

这样前端才能在原有抽屉结构下正确回显当前节点状态。

### 节点状态显示映射

前端展示文案与数据库状态值分离，推荐固定映射如下：

| 数据库状态值 | 前端显示文案 |
|------------|-------------|
| `PENDING` | `待审批` |
| `WAITING` | `待审批` 或沿用现有文案 |
| `APPROVED` | `已通过` |
| `REJECTED` | `已驳回` |
| `WITHDRAWN` | `已撤回` |

## 数据库字段设计口径

以下为本场景中关键数据字段的职责定义。

## 1. Plan 主表

建议以 `sys_plan` 或等价 Plan 表中的这些字段为主：

| 字段 | 含义 | 撤回后目标值 |
|------|------|-------------|
| `id` | Plan 主键 | 不变 |
| `status` | Plan 业务状态 | `DRAFT` |
| `updated_at` | 更新时间 | 更新 |
| `target_org_id` | 目标部门 | 不变 |
| `created_by_org_id` | 发起部门 | 不变 |

说明：

- 页面上的“草稿 / 待审批 / 已下发”最终都应该能回溯到 `plan.status`

## 2. 工作流实例表

以 `audit_instance` 为准：

| 字段 | 含义 | 撤回后目标 |
|------|------|-----------|
| `entity_type` | 实体类型 | `PLAN` |
| `entity_id` | 对应 Plan ID | 不变 |
| `status` | 实例状态 | `IN_REVIEW` |
| `requester_id` | 发起人 | 不变 |
| `started_at` | 发起时间 | 保留 |
| `completed_at` | 结束时间 | 保持未结束 / `NULL` |

说明：

- 当前后端已将 `audit_instance.status` 收敛为三种：
  - `IN_REVIEW`
  - `APPROVED`
  - `REJECTED`
- 撤回不再通过额外实例终态表达，而是继续维持当前实例链路

## 3. 工作流步骤实例表

以 `audit_step_instance` 为准：

| 字段 | 含义 | 撤回后目标 |
|------|------|-----------|
| `instance_id` | 所属流程实例 | 不变 |
| `step_name` | 节点名称 | 保留 |
| `status` | 节点状态 | 当前 `PENDING` 节点改为 `WITHDRAWN` |
| `approver_id` | 审批人 | 保留 |
| `comment` | 审批意见 | 默认补成 `提交人撤回`（若原值为空） |
| `approved_at` | 处理时间 | 补写当前撤回时间 |

说明：

- 当前后端节点状态口径包含：`PENDING / WAITING / APPROVED / REJECTED / WITHDRAWN`
- 节点记录继续保留，供抽屉沿用原有审批链展示
- 不需要因为撤回而新增另一套抽屉模式
- 再次发起时，后端直接复用原实例，并把最新 `WITHDRAWN` 节点恢复回 `PENDING`
- 前端对用户统一显示中文状态，其中 `WITHDRAWN` 显示为 `已撤回`

## 4. 指标表

以 `indicator.status` 为准：

| Plan 状态 | Indicator 状态映射 |
|----------|------------------|
| `DRAFT` | `DRAFT` |
| `PENDING` | `DRAFT` |
| `DISTRIBUTED` | `DISTRIBUTED` |

说明：

- 这也是为什么顶部 Plan 撤回不能调指标撤回接口
- 因为 Plan 进入 `PENDING` 时，指标通常仍是 `DRAFT`

## 推荐实现方案

## 前端

### 必做项

1. 顶部撤回统一调用 `planStore.withdrawPlan(planId)`
2. 撤回成功后仅刷新：
   - `plan list`
   - 当前年度指标列表
3. 页面按钮和可编辑态统一按 Plan 状态重算
4. 审批抽屉沿用原有结构，只修正节点状态映射
5. 不新增新的抽屉展示模式
6. `/indicators` 与 `/distribution` 页面中的审批入口、可编辑态也统一按 Plan 口径收口

### 可选优化项

1. 撤回成功后提示文案改成：
   - `已撤回，Plan 已恢复为草稿状态`
2. 审批抽屉标题与现有 UI 保持一致，不新增额外模式副标题

## 后端

### 必做项

1. `withdrawPlan` 必须先按 `PlanWorkflowSnapshotQueryService` 校验 `canWithdraw`
2. `withdrawPlan` 成功后同步 `Plan` 和指标状态回到草稿口径
3. `withdrawPlan` 成功后，把当前审批实例中的最新 `PENDING` 节点改成 `WITHDRAWN`
4. 再次发起审批时，复用原实例，并把最新 `WITHDRAWN` 节点恢复成 `PENDING`
5. 快照返回继续服务原有抽屉结构，只修正状态判断字段
6. `audit_instance.status` 只保留 `IN_REVIEW / APPROVED / REJECTED`

### 推荐项

1. `withdrawPlan` 响应直接回传：
   - `status = DRAFT`
   - `workflowStatus = IN_REVIEW`
   - `currentStepName / currentApproverId / currentApproverName` 优先返回最新 `WITHDRAWN` 节点信息
   - `canWithdraw = false`

这样前端可以直接沿用原抽屉结构显示 `已撤回`，同时不需要改 UI 结构。

## 验收标准

以下场景全部通过，才算本设计完成。

### 场景 1. 草稿态正常发起

1. 草稿状态下点击 `发起审批`
2. 页面变成 `待审批`
3. 主按钮变 `撤回`
4. 审批入口变 `审批中`
5. 页面不可编辑

### 场景 2. 第一个审批节点前段撤回

1. 发起人提交后，审批仍在允许撤回阶段
2. 点击顶部 `撤回`
3. 撤回成功
4. 页面恢复 `草稿`
5. 主按钮恢复 `发起审批`
6. 审批入口恢复 `审批进度`
7. 页面恢复可编辑

### 场景 3. 撤回后打开审批抽屉

1. 撤回成功后点击审批入口
2. 抽屉继续展示原有审批链
3. 当前最新节点前端显示 `已撤回`
4. 其他历史节点状态继续沿用原有系统口径
5. 抽屉中不存在新的可审批 `PENDING` 节点

### 场景 4. 撤回后再次发起

1. 撤回成功后再次点击 `发起审批`
2. 后端不新建实例，而是复用原审批实例
3. 原来最新的 `已撤回` 节点恢复成 `待审批`
4. 页面重新变成 `待审批`
5. 主按钮重新变成 `撤回`
6. 页面重新进入不可编辑态

### 场景 5. 撤回历史保留

1. 撤回后重新打开审批抽屉
2. 抽屉仍按原有审批链展示
3. 不因为引入新模式而改变现有 UI 结构

### 场景 6. 审批不通过后恢复草稿

1. Plan 已发起审批
2. 审批流程被驳回或不通过
3. 页面状态恢复 `草稿`
4. 顶部主按钮显示 `发起审批`
5. 页面恢复可编辑
6. 不出现 `重新提交` 独立按钮语义

## 结论

本场景的正确设计应当是：

- 顶部撤回 = Plan 级撤回
- 页面编辑态 = Plan 状态驱动
- 审批抽屉 = 沿用原有实例链，只修正状态逻辑
- 节点状态词 = 沿用数据库和原系统已有口径
- 前端节点文案继续显示中文，撤回态显示为 `已撤回`

这样才能满足你要的最终体验：

- 误发起后可撤回
- 撤回后页面回到最初草稿态
- 可继续编辑
- 审批抽屉仍使用原有链路和状态词
- 撤回节点显示 `已撤回`
- 再次发起直接复用原实例继续流转
